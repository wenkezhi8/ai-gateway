## Container Architecture

### Docker Network

All services run in a dedicated Docker network (`ai-gateway-network`):

```
Network: ai-gateway-network (172.28.0.0/16)
в”ңв”Җв”Җ gateway         (172.28.0.10)
в”ңв”Җв”Җ redis           (172.28.0.11)
в”ңв”Җв”Җ web             (172.28.0.12)
в”ңв”Җв”Җ prometheus      (172.28.0.20)
в”ңв”Җв”Җ grafana         (172.28.0.21)
в””в”Җв”Җ alertmanager    (172.28.0.22)
```

### Volume Mapping

```
Docker Volumes
в”ңв”Җв”Җ gateway-data     вҶ’ /app/data (SQLite database)

в”ңв”Җв”Җ prometheus-data  вҶ’ /prometheus (Metrics storage)
в”ңв”Җв”Җ grafana-data     вҶ’ /var/lib/grafana (Dashboards)
в””в”Җв”Җ alertmanager-dataвҶ’ /alertmanager (Alert state)

Host Mounts
в”ңв”Җв”Җ ./configs        вҶ’ /app/configs (Configuration files)
в”ңв”Җв”Җ ./monitoring     вҶ’ /etc/prometheus, /etc/grafana (Monitoring config)
в””в”Җв”Җ ./logs           вҶ’ /app/logs (Log files - optional)
```

## Data Flow

### Request Flow

```
1. User Request
   в””в”Җв–¶ Web Dashboard (http://localhost:3000)
        в”Ӯ
        в”ңв”Җв–¶ Static Assets (React App)
        в”Ӯ
        в””в”Җв–¶ API Requests вҶ’ Gateway API (http://localhost:8000)
             в”Ӯ
             в”ңв”Җв–¶ Rate Limiting Check
             в”Ӯ    в””в”Җв–¶ Redis Cache
             в”Ӯ
             в”ңв”Җв–¶ Route to Provider
             в”Ӯ    в”ңв”Җв–¶ OpenAI API
             в”Ӯ    в”ңв”Җв–¶ Anthropic API
             в”Ӯ    в””в”Җв–¶ Azure OpenAI API
             в”Ӯ
             в”ңв”Җв–¶ Response Caching
             в”Ӯ    в””в”Җв–¶ Redis Cache
             в”Ӯ
             в””в”Җв–¶ Return Response
```

### Monitoring Data Flow

```
1. Metrics Collection
   Gateway API
   в””в”Җв–¶ /metrics endpoint (Prometheus format)
        в”Ӯ
        в””в”Җв–¶ Prometheus scrapes every 15s
             в”Ӯ
             в”ңв”Җв–¶ Store in TSDB
             в”Ӯ
             в””в”Җв–¶ Evaluate alert rules
                  в”Ӯ
                  в””в”Җв–¶ Fire alerts вҶ’ Alertmanager
                       в”Ӯ
                       в”ңв”Җв–¶ Email notifications
                       в”ңв”Җв–¶ Slack webhook
                       в””в”Җв–¶ PagerDuty (optional)

2. Visualization
   Grafana
   в””в”Җв–¶ Query Prometheus
        в”Ӯ
        в””в”Җв–¶ Render dashboards
             в”Ӯ
             в””в”Җв–¶ User views in browser
```

### Security Best Practices

1. **Network Isolation**
   - All services in private Docker network
   - Only necessary ports exposed to host
   - Monitoring endpoints not publicly accessible

2. **API Key Security**
   - Stored in .env file (not in code)
   - .env excluded from git
   - Can use Docker secrets in production

3. **Container Security**
   - Gateway runs as non-root user
   - Read-only config mounts
   - Resource limits enforced

4. **Monitoring Access**
   - Grafana requires authentication
   - Prometheus not exposed externally
   - Alertmanager not exposed externally

## Scaling Considerations

### Vertical Scaling (Increase Resources)

- Increase CPU/memory limits in docker-compose.prod.yml
- Adjust Redis maxmemory
- Increase Prometheus retention

### Horizontal Scaling (Add Instances)

1. **Gateway Scaling**
   - Deploy multiple gateway instances
   - Use load balancer (Nginx/HAProxy)
   - Ensure sticky sessions if needed

2. **Redis Scaling**
   - Use Redis Sentinel for HA
   - Or Redis Cluster for sharding
   - Consider external Redis service

3. **Database Scaling**
   - Move from SQLite to PostgreSQL
   - Set up read replicas
   - Consider managed database service
