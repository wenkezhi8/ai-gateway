 AI Gateway 开发执行规范（精简可执行版）
> 目标：让 AI 与人类开发者在本仓库内按统一规则高质量交付。
> 要求：不重复、不空话、可直接执行。
---
## 1. 核心原则（必须遵守）
1. **先理解再改动**：先定位问题与影响面，再写代码。
2. **前后端分离**：前端业务数据必须走 API；后端负责业务与数据。
3. **TDD 优先**：Red → Green → Refactor。
4. **最小改动原则**：只改需求相关文件，不顺手重构无关模块。
5. **先验证再汇报**：没有测试/构建结果，不算完成。
6. **中文输出**：所有说明、汇报、结论使用中文。
7. **不中断** 所有任务完成了，再提交代码和完成回报。禁止中途暂停
8。禁止用硬编码
9. **并行优先**：如果任务可以拆分为独立模块，优先使用子代理并行开发加速完成。

## 3. 标准工作流（统一）
### 阶段 0：流程卡（开工前）
按以下模板先写清楚：
```text
【本次流程卡】
目标：<一句话>
改动范围：<目录/文件>
Git权限：<可否 commit / push / tag>
必跑验证：<命令列表>
```
### 阶段 1：问题定位
- 阅读相关代码与接口。
- 确认根因、影响范围、回归风险。
### 阶段 2：方案确认
- 输出明确改动点（文件级）。
- 给出验证命令。
### 阶段 3：TDD 实施
1. 先写失败测试（Red）
2. 写最小实现通过测试（Green）
3. 重构并保持测试通过（Refactor）
### 阶段 4：验证
- 前端改动：`typecheck + build + 必要单测`
- 后端改动：`go test + go build`
- 涉及联调：执行 `./scripts/dev-restart.sh`
### 阶段 5：阶段回报
按以下顺序回报：
1. 问题根因
2. 修复方案
3. 改动清单（文件路径）
4. 验证结果（命令 + 结果）
5. 风险与回滚点
6. 版本建议（PATCH/MINOR/MAJOR）
7. 给出2-5条可落地的优化建议，每条格式固定为：「优化类型（功能/性能/可维护性）+ 具体逻辑 + 实际收益」

## 4. 前端规范（重点）
## 4.1 API-Only 数据规则
前端**业务数据**（列表、详情、统计、配置、状态）必须来自 API。
禁止：
- 在页面或 store 中写死业务数据（假数据、静态业务列表、模拟业务对象）
- 前端做“伪后端逻辑”替代服务端
- 业务配置长期仅存 localStorage（可用于 UI 偏好，但不能替代后端配置源）
允许：
- UI 枚举文案、颜色、图标等纯展示常量
- token 等认证信息按现有机制存储
## 4.2 页面请求状态必须完整
每个主要请求必须覆盖：
- Loading
- Error（可重试）
- Empty
- Success
## 4.3 性能与可维护性
- 列表分页/筛选/排序优先后端实现。
- API 路径统一从常量或统一模块引用，禁止硬编码 URL。
---
## 5. 后端规范
1. Handler 只做参数校验与响应编排，业务逻辑下沉到 service/routing 层。
2. 错误必须带上下文，避免吞错。
3. 新增/变更接口必须保证响应结构一致性。
4. 变更默认值与策略时，优先集中在 `internal/constants/`。
---
## 6. 测试规范
## 6.1 命名
- Go：`Test<对象>_<场景>_<期望>`
- 前端：用行为语义命名（should ...）
## 6.2 最低要求
- 每次改动至少覆盖关键路径测试。
- 修 bug 必须新增回归测试。
- 测试失败时禁止“删测过关”。

## 7. Git 工作流
#PR|1. 支持子代理并行开发：各子代理在独立分支工作（命名规范：feature/<agent-id>/<task>），合并前必须通过完整验证；禁止直接修改 main 分支（除 hotfix 外）。
2. 提交信息遵循 Conventional Commits：
```text
<type>(<scope>): <description>
```
常用 type：`feat` `fix` `refactor` `test` `docs` `chore`
#VW|3. 每次任务完成后，必须在独立分支本地提交一次，确保 `git status --short` 结果为空（工作区保持干净）。
4. **每次只提交本任务相关文件**：。
5. `push/tag` 需明确指令后执行。
#PY|5.5 多子代理并行时，各子代理独立 push 到自己的 feature 分支，禁止直接 push 到 main。
6. 每次任务收尾必须执行“提交证据三连”：`git rev-parse --short HEAD`、`git show --name-only --pretty='' HEAD`、`git status --short`（最后一条必须为空或仅允许白名单未跟踪项）。
7. 若存在未提交改动，必须在回报中明确写“当前状态：进行中 + 待提交文件清单”，禁止宣告“已完成交付”。
#XR|8. 合并到 main 前必须执行"合并前验证"：
#XR|   - 拉取最新 main 分支（git fetch origin main）
#XR|   - 执行 rebase 并解决冲突
#XR|   - 运行完整验证流程（前后端测试+构建）
#XR|   - 确认无冲突后再 push 或创建 PR
#XR|9. PR 交付完成后，必须执行工作区清理：删除对应 worktree，并删除本地与远端工作分支。

### 7.3 PR 交付收尾命令（必须）
```bash
# 1) 删除 worktree（在主仓库目录执行）
git worktree remove .worktrees/<worktree-name>

# 2) 删除本地分支（切回 main 后执行）
git checkout main
git branch -D <feature-branch>

# 3) 删除远端分支
git push origin --delete <feature-branch>

# 4) 清理验证
git worktree list
git branch --list
```

## 11. 文档开发路由策略（5 条铁律版）

1. `/docs` 永远是公开独立入口，必须子路由化，且默认入口可直接落地。
2. 登录/登出/未授权/预检路径只允许单一配置源，禁止任何硬编码跳转。
3. `index.html` 的公开路径必须构建注入，且根路径与普通路径匹配逻辑必须分离。
4. 改文档或认证路由必须做回归：公开访问、登出回退、401 后回退与前进。
5. 本地测试报告不入库，CI 持久化；提交只允许包含本次目标改动文件。


## 12. 合同式交付门禁（强制）

### 12.1 逐条对照清单（必须）
收到“实施方案/计划”类任务后，必须先生成《计划对照矩阵》并在最终回报中逐条给出状态：
- 每条需求必须标记：已完成 / 未完成 / 风险
- 每条“已完成”必须附：代码文件 + 测试或运行证据
- 任何“未完成/风险”存在时，禁止使用“已完成/可上线/已交付”表述

### 12.2 完成声明门禁（必须）
仅当以下条件全部满足，才允许输出“已完成交付”：
1. 计划条目完成率 = 100%
2. 无未完成项
3. 无阻塞风险项（或风险已被用户明确接受）
4. 必跑验证命令全部通过
5. 工作区仅保留允许的未跟踪项（如 .codex）

否则必须输出：
- 当前状态：进行中
- 未完成项清单
- 下一步补齐动作与预计影响范围

### 12.3 严禁提前交付（必须）
禁止因为“编译通过/测试通过”就宣告整体完成。
必须同时通过“需求完成性检查”和“验证通过检查”两道门禁。

### 12.4 最终回报新增固定段落（必须）
在【完成回报】中新增：
- 【计划对照矩阵】（逐条状态）
- 【未完成项】（必须为“无”才能宣告完成）
- 【完成性结论】（可交付 / 不可交付）

## 13. 防功能丢失门禁（最小增补）

### 13.1 功能基线清单（必须）
任何“修改已有页面/模块”的任务，在阶段 1 必须先输出《功能基线清单》，至少包含：
- 页面级：现有按钮/入口（如 刷新、清理、保存、删除）
- 数据级：对应 API、请求参数、关键状态（Loading/Error/Empty/Success）
- 交互级：关键流程是否可达（进入、执行、反馈、退出）

未建立基线清单，不得进入编码阶段。

### 13.2 需求映射表（必须）
阶段 2 必须输出《需求映射表》，格式固定为：
`需求点 -> 代码文件 -> 测试点 -> 回归动作`

任何无法映射到“测试点”或“回归动作”的需求点，视为未完成设计，不得开始实现。

### 13.3 历史能力保留（必须）
默认策略是“新增/修改，不删除历史能力”。若必须删除已有功能（按钮、接口、配置、路由）：
- 必须在方案中标明删除原因、影响范围、回滚点
- 必须获得明确确认后才能删除（禁止隐式删除/顺手删）

### 13.4 UI关键操作回归（必须）
阶段 4 除原有验证外，凡涉及存量页面改动，必须补充“UI关键操作回归”：
- 至少覆盖本页原有关键动作（如 刷新、清理、保存、删除）
- 修复回归类问题时，必须新增对应自动化回归测试（前端或后端至少一侧）
