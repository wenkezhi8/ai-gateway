 AI Gateway 开发执行规范（精简可执行版）
> 目标：让 AI 与人类开发者在本仓库内按统一规则高质量交付。
> 要求：不重复、不空话、可直接执行。
---
## 1. 核心原则（必须遵守）
1. **先理解再改动**：先定位问题与影响面，再写代码。
2. **前后端分离**：前端业务数据必须走 API；后端负责业务与数据。
3. **TDD 优先**：Red → Green → Refactor。
4. **最小改动原则**：只改需求相关文件，不顺手重构无关模块。
5. **先验证再汇报**：没有测试/构建结果，不算完成。
6. **中文输出**：所有说明、汇报、结论使用中文。
---
## 2. 快速命令
### 后端
```bash
make lint
make test
make build
go build ./cmd/gateway
```
### 前端
```bash
cd web && npm run typecheck
cd web && npm run build
cd web && npm run test:unit
```
### 联调重启
```bash
./scripts/dev-restart.sh
```

## 3. 标准工作流（统一）
### 阶段 0：流程卡（开工前）
按以下模板先写清楚：
```text
【本次流程卡】
目标：<一句话>
改动范围：<目录/文件>
Git权限：<可否 commit / push / tag>
必跑验证：<命令列表>
```
### 阶段 1：问题定位
- 阅读相关代码与接口。
- 确认根因、影响范围、回归风险。
### 阶段 2：方案确认
- 输出明确改动点（文件级）。
- 给出验证命令。
### 阶段 3：TDD 实施
1. 先写失败测试（Red）
2. 写最小实现通过测试（Green）
3. 重构并保持测试通过（Refactor）
### 阶段 4：验证
- 前端改动：`typecheck + build + 必要单测`
- 后端改动：`go test + go build`
- 涉及联调：执行 `./scripts/dev-restart.sh`
### 阶段 5：阶段回报
按以下顺序回报：
1. 问题根因
2. 修复方案
3. 改动清单（文件路径）
4. 验证结果（命令 + 结果）
5. 风险与回滚点
6. 版本建议（PATCH/MINOR/MAJOR）
7. 给出2-5条可落地的优化建议，每条格式固定为：「优化类型（功能/性能/可维护性）+ 具体逻辑 + 实际收益」
---
## 4. 前端规范（重点）
## 4.1 API-Only 数据规则
前端**业务数据**（列表、详情、统计、配置、状态）必须来自 API。
禁止：
- 在页面或 store 中写死业务数据（假数据、静态业务列表、模拟业务对象）
- 前端做“伪后端逻辑”替代服务端
- 业务配置长期仅存 localStorage（可用于 UI 偏好，但不能替代后端配置源）
允许：
- UI 枚举文案、颜色、图标等纯展示常量
- token 等认证信息按现有机制存储
## 4.2 页面请求状态必须完整
每个主要请求必须覆盖：
- Loading
- Error（可重试）
- Empty
- Success
## 4.3 性能与可维护性
- 列表分页/筛选/排序优先后端实现。
- API 路径统一从常量或统一模块引用，禁止硬编码 URL。
---
## 5. 后端规范
1. Handler 只做参数校验与响应编排，业务逻辑下沉到 service/routing 层。
2. 错误必须带上下文，避免吞错。
3. 新增/变更接口必须保证响应结构一致性。
4. 变更默认值与策略时，优先集中在 `internal/constants/`。
---
## 6. 测试规范
## 6.1 命名
- Go：`Test<对象>_<场景>_<期望>`
- 前端：用行为语义命名（should ...）
## 6.2 最低要求
- 每次改动至少覆盖关键路径测试。
- 修 bug 必须新增回归测试。
- 测试失败时禁止“删测过关”。
---
## 7. Git 工作流
1. 默认在 `main` 开发（如需分支由任务指定）。
2. 提交信息遵循 Conventional Commits：
```text
<type>(<scope>): <description>
```
常用 type：`feat` `fix` `refactor` `test` `docs` `chore`
3. 每次任务完成后，必须本地提交一次，确保 `git status --short` 结果为空（工作区保持干净）。
4. `push/tag` 需明确指令后执行。
---
## 8. 变更完成检查清单（交付前）
- [ ] 改动范围与需求一致
- [ ] 无新增业务硬编码数据
- [ ] TDD 记录完整（至少一条 Red→Green 证据）
- [ ] 前端：typecheck/build 通过
- [ ] 后端：go test/build 通过
- [ ] 输出了风险与回滚点
