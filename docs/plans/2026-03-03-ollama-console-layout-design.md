# Ollama 控制台改版设计（上下双面板 + Tab 混合）

## 1. 目标

- 将现有三标签页结构调整为单页双面板结构，减少来回切页成本。
- 上方面板固定承担 Ollama 服务与模型资产管理。
- 下方面板通过 Tab 承担专项配置，分别为意图路由与向量管理。
- 清晰划分职责：模型下载/删除只在服务管理面板；意图路由与向量管理只负责切换模型和启动模型。

## 2. 信息架构与布局

页面从上到下分为两个主面板：

1. `服务管理面板（始终可见）`
   - 基础服务控制：安装、启动、停止、状态刷新、轮询设置。
   - 统一模型管理：模型输入、下载、删除、常用模型入口、本地模型列表、运行中模型列表。
   - 不再承载意图分类器开关与向量 Pipeline 开关。

2. `高级配置面板（Tab）`
   - Tab A：意图路由。
   - Tab B：向量管理。

## 3. 职责边界

### 3.1 Ollama 服务管理面板

- 保留“模型资产”相关全量操作：下载、删除、查看。
- 保留服务生命周期操作：安装、启动、停止。
- 作为全局模型来源，向下方两个 Tab 提供可切换模型候选。

### 3.2 意图路由 Tab

- 负责意图分类器功能配置与运行观测。
- 新增或强化“仅意图域内”的模型操作：
  - 切换当前分类模型。
  - 启动选定分类模型。
- 不提供下载/删除能力，避免与服务管理面板重复。

### 3.3 向量管理 Tab

- 负责向量 Pipeline 配置、索引状态、缓存策略可视化。
- 新增或强化“仅向量域内”的模型操作：
  - 切换当前 embedding 模型。
  - 启动选定 embedding 模型。
- 不提供下载/删除能力，避免与服务管理面板重复。

## 4. 交互与数据流

- 页面初始化：统一加载服务状态、模型状态、意图配置、向量配置。
- 上方面板模型列表作为下方两 Tab 的可选模型集合来源。
- 下方 Tab 触发“切换模型/启动模型”后，回流刷新上方面板运行中模型信息，保证跨面板一致。
- 失败时展示明确错误信息并提供重试入口，不静默失败。

## 5. 异常与状态设计

- 各主面板继续覆盖 `loading / error / empty / success` 四态。
- 启动模型失败时在对应 Tab 就地提示，不跳转。
- 当模型未下载时，Tab 中启动动作应给出“请先到服务管理下载模型”的引导文案。

## 6. 测试策略

- 更新 `web/src/views/ollama/index.test.ts`：验证双面板结构与 Tab 存在。
- 新增或调整服务管理测试：确认模型下载/删除只在上方面板。
- 新增或调整意图路由、向量管理测试：
  - 验证仅存在切换模型/启动模型操作。
  - 验证无下载/删除入口。
- 保持 `npm run test:unit` 与 `npm run typecheck` 通过。

## 7. 兼容性与风险

- 主要风险是职责迁移后用户短期操作路径变化；通过面板标题与提示文案降低学习成本。
- 由于复用既有 composable 与 API，后端接口风险低。
- 若出现回归，可回滚到原标签页版本的 `index.vue` 结构变更。
