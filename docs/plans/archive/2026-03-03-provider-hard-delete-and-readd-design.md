# 服务商真实删除与可重加设计

## 背景与问题

- 当前模型管理页删除服务商只删除了模型评分与默认模型映射，页面刷新后会再次从 `/api/v1/config/providers` 合并服务商数据，导致“删除后复活”。
- 用户期望：
  - 所有服务商都可删除；
  - 删除后刷新、重启都不恢复；
  - 删除后仍可在“添加服务商”中重新添加；
  - 添加列表包含 `web/public/logos` 中服务商，并尽可能展示中文名，缺失时英文 ID 兜底。

## 目标

- 提供“真实删除”能力（级联清理 + 持久化），杜绝刷新/重启复活。
- 提供“服务商目录”作为可重加来源，确保删除后可恢复添加。

## 非目标

- 不新增复杂权限模型。
- 不引入新的数据库表。
- 不恢复旧版“模型评分编辑 UI”。

## 总体方案

### 1) 后端：删除改为级联真实删除

扩展 `DELETE /api/admin/providers/:id` 的语义：

1. 从运行时 provider registry 删除目标服务商。
2. 级联删除关联账号（同时匹配 `Provider` 与 `ProviderType`），并持久化 `data/accounts.json`。
3. 清理 Smart Router 中该服务商相关模型记录（`/admin/router/models` 的底层数据），并保留现有持久化行为。
4. 清理 provider defaults 中对应条目并持久化。
5. 更新 `configs/config.json` 的 `providers` 列表，移除目标服务商，保证重启后不恢复。

返回删除结果统计字段，便于前端展示实际清理范围。

### 2) 前端：允许删除全部服务商

- 去掉“仅 custom 可删除”的限制；
- 删除动作直接调用统一删除接口，不再由前端手工编排多步删除；
- 删除成功后本地列表与后端状态保持一致。

### 3) 前端：服务商目录 + 可重加

- 新增“服务商目录”常量，来源为 `web/public/logos/*.svg`；
- 目录项字段：`id`、`label`、`color`、`logo`；
- `label` 优先中文映射（已知服务商），未知使用英文 ID 兜底；
- “添加服务商”下拉数据源改为“公开服务商 + 服务商目录”合并去重。

## 数据流

1. 页面加载：
   - 拉取公开服务商；
   - 合并本地服务商目录用于“添加服务商”候选。
2. 删除服务商：
   - 前端调用 `DELETE /api/admin/providers/:id`；
   - 后端完成级联清理并持久化；
   - 前端刷新列表。
3. 重新添加：
   - 从下拉选择目录中的服务商；
   - 沿用现有新增流程写回路由模型与默认模型。

## 异常与容错

- 删除过程中任一步失败时返回明确错误并停止“成功提示”。
- 若 `configs/config.json` 不可写，删除应整体失败并提示运维修复权限。
- 中文名映射不存在时回退英文 ID，保证功能不中断。

## 测试策略

### 后端

- 新增 provider 删除接口回归测试：
  - 断言 registry 移除；
  - 断言关联账号删除并落盘；
  - 断言 provider defaults 与模型记录被清理；
  - 断言 `configs/config.json` 已移除 provider。

### 前端

- 更新模型管理页单测：
  - 所有服务商可删除；
  - 添加弹窗下拉包含目录服务商；
  - 删除后仍可通过下拉重加。

## 风险与回滚

- 风险：真实删除会影响正在使用中的路由能力。
- 缓解：删除确认弹窗增加明确提示，并展示将级联删除账号数量。
- 回滚：可通过“添加服务商 + 重新配置账号”恢复；必要时从版本管理恢复 `configs/config.json` 与 `data/accounts.json`。
